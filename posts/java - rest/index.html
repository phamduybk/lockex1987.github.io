REST

Seriously, what is REST? Why always REST?

REST (REpresentational State Transfer) is an software architectural style of the World Wide Web.

Architectural constraints (5)

  • Client-Server
    • Portability of client
    • Focus on simple server
  • Stateless
    • Phi trạng thái
    • Dễ scale
    • Store authentication in database, not session?
  • Cacheable
    • Responses must define themselves as cacheable or not, implicitly or explicitly.
  • Layered system
    • A client cannot ordinarily tell whether it is connected directly to the end server, or to an intermediary along the way.
    • Intermediary servers may improve system scalability by enabling load balancing and by providing shared caches.
    • They may also enforce security policies.
  • Uniform interface
    • Identification of resources
    • Manipulation of resources through these representations
    • Self-descriptive messages
    • Hypermedia as the engine of application state (HATEOAS)

Applied to web services

  • Base URI
    • Hiển thị cấu trúc thư mục như URls
    • Hệ thống
  • An Internet media type (JSON)
    • Khả chuyển, đa nền tảng
  • Standard HTTP methods
    • Sử dụng các phương thức HTTP một cách rõ ràng (GET: read, POST: insert or update, DELETE: delete, PUT: partial update like mark operation)
    • Logic
  • Hypertext links to reference state
  • Hypertext links to reference-related resources

REST vs SOAP vs WSDL

REST is an architectural style.

SOAP (Simple Object Access Protocol) is a protocol.

WSDL (Web Service Description Language) is an XML document that describes a web service.
